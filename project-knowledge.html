Kiến thức & kiến trúc chính dùng trong dự án

File này tóm tắt các kiến thức, công nghệ, cách tổ chức code, validate và các best practice hiệu năng đang dùng trong hệ thống Crowdsourced Waste Collection & Recycling System. Mục tiêu là để dev mới có thể đọc nhanh và nắm bức tranh tổng quan.

1. Công nghệ & stack

  • Backend Java + Spring Boot
  • ORM Spring Data JPA / Hibernate (code-first, entity <-> bảng DB)
  • DB Quan hệ (MySQL / tương đương; dùng index tối ưu query)
  • Auth JWT + Token denylist (bảng invalidated_tokens + cache in-memory)
  • Storage Cloudinary để lưu ảnh (báo cáo rác, voucher banner/logo)
  • Scheduling Spring @Scheduled cho các job tự động (timeout, SLA)

2. Các module chức năng chính

2.1. Báo cáo rác của công dân (WasteReport)

  • Entity: WasteReport map với bảng waste_reports.
  • Service chính: WasteReportServiceImpl.
  • Repository: WasteReportRepository với một số query thống kê.

Luồng create/update report:

  • Validate input (ảnh, toạ độ, category, số lượng, loại file) trong validateCreateRequest.
  • Chống spam report gần nhau (khoảng 30m, trong 10 phút) bằng query countRecentNearDuplicate.
  • Tạo bản ghi WasteReport, WasteReportItem (chi tiết loại rác).
  • Upload ảnh lên Cloudinary qua CloudinaryService, lưu URL vào WasteReport & ReportImage.
// Ví dụ mapping entity cơ bản (WasteReport)
@Entity
@Table(
    name = "waste_reports",
    indexes = {
        @Index(name = "idx_wr_citizen_status_created", columnList = "citizen_id,status,created_at"),

public class WasteReport {

    private Integer id;

    @ManyToOne(fetch = FetchType.LAZY)

2.2. Yêu cầu thu gom (CollectionRequest) & tác vụ của Collector

  • Entity: CollectionRequest map với bảng collection_requests.
  • Service:
    • CollectorServiceImpl: tác vụ cho nhân viên thu gom (task list, history, update status,...).
    • EnterpriseAssignmentService, TaskAutomationServiceImpl: tự động phân công & job SLA.
  • Repository: CollectionRequestRepository với nhiều projection view phục vụ dashboard và task list.

Kiến thức quan trọng:

  • Dùng enum CollectionRequestStatus và WasteReportStatus để quản lý state máy trạng thái.
  • Dùng projection interface (vd CollectorTaskView, CollectorWorkHistoryView) để tránh N+1 và giảm dữ liệu trả về.
  • Nhiều phương thức cập nhật trạng thái dùng JPQL update trả về số dòng cập nhật để chống race condition (kiểm tra trạng thái mong đợi).

2.3. Điểm thưởng & lịch sử điểm (PointTransaction)

  • Entity: PointTransaction map bảng point_transactions.
  • Repository: PointTransactionRepository với query tổng điểm, filter theo ngày, enterprise, loại giao dịch.
  • Service: WasteReportServiceImpl (lịch sử thưởng người dân), EnterpriseStatisticsServiceImpl (thống kê enterprise).

Best practice đã dùng:

  • Không filter bằng vòng lặp Java với list lớn: lịch sử điểm dùng query findByCitizenIdAndDateRange.
  • Dùng aggregate SQL (SUM, GROUP BY) cho thống kê thay vì tính toán Java.
  • Thêm index trên citizen_id,transaction_type,created_at để tối ưu query theo user + thời gian.

2.4. Khiếu nại / Feedback (Feedback)

  • Entity: Feedback map bảng feedbacks.
  • Repository: FeedbackRepository với @EntityGraph để tránh N+1 khi xem danh sách.
  • Service:
    • WasteReportServiceImpl: công dân gửi complaint.
    • EnterpriseFeedbackServiceImpl: doanh nghiệp xem & xử lý feedback.
    • AdminComplaintServiceImpl: admin xem tất cả khiếu nại.

3. Validate & rule nghiệp vụ tiêu biểu

3.1. Validate báo cáo rác (createReport / updateReport)

  • Ảnh: không được rỗng, phải là MIME type image/*, giới hạn kích thước tối đa (10MB input, nén xuống < 1MB trước khi upload).
  • Danh sách loại rác: không rỗng, số lượng quantities phải khớp số category, quantity >= 0.
  • Toạ độ: bắt buộc, nằm trong range hợp lệ (-90 ≤ lat ≤ 90, -180 ≤ lng ≤ 180).
  • Chống spam: nếu công dân vừa gửi report ở gần vị trí đó (±0.00030 độ, ~30m) trong vòng 10 phút thì reject.
  • Update/hủy report: chỉ cho phép khi status == PENDING và chưa có CollectionRequest liên quan.

3.2. Rule SLAs & tự động hoá

  • Job checkAssignedTasksTimeout: nếu collector không nhận task trong X giờ:
    • Tăng số lần vi phạm của collector.
    • Huỷ phân công, trả task về trạng thái enterprise (ACCEPTED_ENTERPRISE).
    • Ghi log vào CollectionTracking.
  • Job checkSlaViolations: nếu thời gian xử lý vượt SLA:
    • Đánh dấu slaViolated = true cho task.
    • Tăng violation count collector và có thể chuyển collector sang trạng thái SUSPEND khi quá ngưỡng.

3.3. Rule khiếu nại (complaints/feedback)

  • Chỉ cho phép các loại complaint chuẩn hoá: COMPLAINT_COLLECTION, COMPLAINT_REWARD, COMPLAINT_SYSTEM.
  • Khiếu nại về thưởng (COMPLAINT_REWARD) chỉ hợp lệ nếu báo cáo đã ở trạng thái COLLECTED.
  • Enterprise/Admin giải quyết complaint:
    • Cập nhật status và resolution của feedback.
    • Nếu xác nhận complaint đúng (status = RESOLVED) thì tăng violation cho collector.

4. Bảo mật (Security)

4.1. Xác thực & uỷ quyền (AuthZ/AuthN)

  • JWT:
    • Client đăng nhập (qua AuthServiceImpl) nhận về access token (JWT) + có thể refresh token.
    • JWT chứa claim sub (email), roles, jti (ID duy nhất của token), exp (hết hạn).
  • Spring Security:
    • Cấu hình trong SecurityConfig:
      • Filter JWT đọc token từ header Authorization: Bearer ....
      • Phân quyền dựa trên role (ADMIN / ENTERPRISE / COLLECTOR / CITIZEN).
  • Token denylist:
    • Bảng invalidated_tokens + service TokenDenylistService dùng để thu hồi token khi logout/ban.
    • Check mỗi request: nếu jti tồn tại trong denylist và chưa hết hạn → token bị từ chối.

4.2. Bảo vệ dữ liệu & input

  • Validate mạnh ở tầng service:
    • Không tin dữ liệu từ client: luôn check trạng thái, quyền sở hữu, loại dữ liệu.
    • Ví dụ: chỉ owner report mới được update/hủy; enterprise chỉ xem được report trong khu vực/service area của mình.
  • Tránh lộ dữ liệu nhạy cảm trong log:
    • CollectionRequest dùng @ToString(exclude = { "report", "enterprise", "collector" }) để log không in tràn nội dung liên quan.
  • Upload file:
    • Chỉ cho phép các extension ảnh: jpg, jpeg, png, gif, bmp (check bằng FileUpLoadUtil.assertAllowedImage).
    • Giới hạn kích thước file, kiểm tra MIME type bắt đầu bằng image/.

4.3. Một số lưu ý bảo mật khác

  • Không trả về thông tin nội bộ khi lỗi: dùng AppException và ErrorCode để chuẩn hoá message.
  • Kiểm tra chặt chẽ quyền truy cập:
    • Enterprise chỉ xem/giải quyết feedback thuộc enterprise của mình (check feedback.collectionRequest.enterprise.id).
    • Collector chỉ xem task thuộc về mình (findByIdAndCollector_Id hoặc kiểm tra lại trong handleUpdateError).

5. API tiêu biểu & luồng nghiệp vụ

5.1. Luồng báo cáo rác của công dân

  1. Công dân tạo report
    • API: POST /api/citizen/reports (request: CreateWasteReportRequest).
    • Service: WasteReportServiceImpl.createReport.
    • Bước:
      1. Lấy citizen từ email trong JWT (requireCitizenByEmail).
      2. Validate input (ảnh, category, quantity, toạ độ) bằng validateCreateRequest.
      3. Check spam bằng wasteReportRepository.countRecentNearDuplicate.
      4. Tạo WasteReport + WasteReportItem.
      5. Upload ảnh → Cloudinary → lưu link.
      6. Trả về WasteReportResponse với thông tin cơ bản + ảnh + category.
  2. Enterprise xem & duyệt report
    • API: GET /api/enterprise/reports (filter theo status).
    • Service: EnterpriseWasteReportServiceImpl.getReports.
    • Bước:
      1. Xác thực enterprise (id từ JWT hoặc param) qua validateEnterprise.
      2. Lấy list report theo trạng thái, sort theo createdAt.
      3. Lọc theo khu vực mà enterprise phục vụ (dùng AddressMatchUtil.isInServiceArea).
      4. Load thêm ảnh, items, liên kết với CollectionRequest để build EnterpriseWasteReportResponse.
  3. Enterprise chấp nhận / từ chối report
    • API: POST /api/enterprise/reports/{id}/accept, /reject.
    • Service: EnterpriseWasteReportServiceImpl.acceptReport / rejectReport.

5.2. Luồng nhiệm vụ của Collector

  1. Lấy danh sách task
    • API: GET /api/collector/tasks (có thể filter theo status / all).
    • Service: CollectorServiceImpl.getTasks.
    • Repository: CollectionRequestRepository.findTasksForCollector... trả về projection CollectorTaskView.
  2. Collector nhận task / bắt đầu / hoàn thành / từ chối
    • API:
      • POST /api/collector/tasks/{id}/accept
      • POST /api/collector/tasks/{id}/start
      • POST /api/collector/tasks/{id}/complete
      • POST /api/collector/tasks/{id}/reject
    • Service: CollectorServiceImpl.acceptTask, startTask, completeTask, rejectTask.
    • Rule:
      • Chỉ đổi trạng thái nếu trạng thái hiện tại đúng mong đợi (ASSIGNED → ACCEPTED_COLLECTOR → ON_THE_WAY → COLLECTED).
      • Kiểm tra quyền sở hữu task (task phải thuộc collector đang đăng nhập).
      • Update đồng thời CollectionRequest và WasteReport.status, ghi log vào CollectionTracking.
  3. Lịch sử làm việc & thống kê
    • API: GET /api/collector/work-history, /stats, /waste-volume...
    • Service: CollectorServiceImpl.getWorkHistory, getStats, getWasteVolumeStats, getWasteTypeStats.
    • Tất cả thống kê sử dụng aggregate SQL (COUNT, SUM, GROUP BY) qua CollectionRequestRepository, CollectorReportItemRepository.

5.3. Luồng điểm thưởng & leaderboard công dân

  1. Lịch sử điểm của công dân
    • API: GET /api/citizen/points/history (có thể filter theo khoảng thời gian).
    • Service: WasteReportServiceImpl.getRewardHistory.
    • Validate:
      • Lấy citizen từ email, nếu không tồn tại → ErrorCode.USER_NOT_EXISTED.
      • Nếu có startDate/endDate: dùng query filter theo createdAt để không load toàn bộ lịch sử.
  2. Tổng điểm & thống kê theo tháng/quý/năm
    • API: GET /api/citizen/points/summary.
    • Service: WasteReportServiceImpl.getMyPointSummary.
    • Rule:
      • Validate year, quarter, month trong range hợp lệ, nếu sai trả HTTP 400.
      • Dùng query sumPointsByCitizenIdAndTypeAndRange với điều kiện thời gian đã chuẩn hoá.
  3. Leaderboard công dân
    • API: GET /api/citizen/leaderboard (filter theo ngày/tháng/năm hoặc khu vực).
    • Repo: PointTransactionRepository.findCitizenLeaderboard (GROUP BY, SUM(points)).

6. Tối ưu hiệu năng & tránh N+1

6.1. Nén & upload ảnh

  • Nén ảnh: FileUpLoadUtil.compressImage
    • Nếu file <= 1MB: giữ nguyên bytes.
    • Nếu lớn hơn: đọc vào BufferedImage, resize 1 lần về tối đa 1920x1080 rồi thử nén với một vài mức quality (0.85 → 0.7 → 0.55 → 0.4).
    • Nếu vẫn >1MB: fallback resize mạnh hơn về 800x800, nén ~0.6.
  • Upload ảnh Cloudinary: WasteReportServiceImpl.saveReportImages
    • Dùng ExecutorService + CompletableFuture để upload song song (tối đa 4 ảnh cùng lúc).
    • Ảnh đầu tiên vẫn là ảnh đại diện; tất cả ảnh lưu vào bảng ReportImage.

6.2. Tránh N+1 bằng EntityGraph & projection

  • WasteReportRepository: findAll(Sort), findByStatus(...) dùng @EntityGraph("citizen","citizen.user") để không N+1 khi lấy tên/email người báo cáo.
  • FeedbackRepository: findAll, findByEnterpriseId fetch sẵn citizen, citizen.user, collectionRequest, collectionRequest.collector, collectionRequest.enterprise.
  • PointTransactionRepository: findByCitizenId, findByCitizenIdAndDateRange fetch sẵn citizen để mapper dùng thông tin công dân không sinh thêm query.
  • CollectionRequestRepository: nhiều query trả về projection interface (view) thay vì entity, giúp tránh lazy-loading & N+1 cho task list / history / leaderboard.

6.3. Index ở cấp DB

  • waste_reports: index theo citizen_id,status,created_at, status,created_at.
  • collection_requests: index theo collector_id,status, enterprise_id,status,created_at, status,assigned_at, sla_violated,started_at, report_id.
  • point_transactions: index theo citizen_id,created_at, citizen_id,transaction_type,created_at.
  • invalidated_tokens: index theo expiryTime hỗ trợ dọn dẹp.
  • feedbacks: index theo citizen_id,created_at, status,created_at.
  • vouchers: index theo active,id, title.

7. Auth & Token Denylist

  • Bảng invalidated_tokens lưu id (jti) + expiryTime cho token bị thu hồi.
  • Service TokenDenylistService:
    • Dùng ConcurrentHashMap làm cache in-memory (max 50k entry) với TTL dương/âm để giảm hit DB.
    • Khi logout hoặc revoke: ghi vào DB + update cache.
    • Khi check: ưu tiên lấy từ cache, miss thì hỏi DB rồi update cache.

8. Hướng phát triển / tối ưu tiếp

  • Bổ sung phân trang cho các API hiện đang trả full list (complaints, feedbacks, voucher list, một số task list).
  • Có thể thêm Redis cho:
    • Cache denylist JWT (thay local cache, chia sẻ giữa nhiều instance).
    • Cache thống kê dashboard (enterprise, collector, citizen leaderboard) với TTL ngắn.
  • Bổ sung test cho các rule quan trọng: chống spam report, SLA, complaint/resolution.

Gợi ý: khi có dev mới join, hãy cho họ đọc file này trước, sau đó mở các class: WasteReportServiceImpl, CollectorServiceImpl, EnterpriseStatisticsServiceImpl, TaskAutomationServiceImpl để nắm kỹ hơn từng luồng nghiệp vụ.
