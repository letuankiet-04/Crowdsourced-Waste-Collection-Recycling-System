Kiến thức & kiến trúc chính dùng trong dự án

File này tóm tắt các kiến thức, công nghệ, cách tổ chức code, validate và các best practice hiệu năng đang dùng trong hệ thống Crowdsourced Waste Collection & Recycling System. Mục tiêu là để dev mới có thể đọc nhanh và nắm bức tranh tổng quan.

1. Công nghệ & stack

  • Backend Java + Spring Boot
  • ORM Spring Data JPA / Hibernate (code-first, entity <-> bảng DB)
  • DB Quan hệ (MySQL / tương đương; dùng index tối ưu query)
  • Cache Redis (Spring Data Redis + Lettuce, dùng làm cache cho dashboard/thống kê Collector và lưu token tạm thời)
  • Auth JWT + Token denylist (bảng invalidated_tokens + cache in-memory, có thể nâng cấp dùng Redis)
  • Storage Cloudinary để lưu ảnh (báo cáo rác, voucher banner/logo)
  • Scheduling Spring @Scheduled cho các job tự động (timeout, SLA)

2. Các module chức năng chính

2.1. Báo cáo rác của công dân (WasteReport)

  • Entity: WasteReport map với bảng waste_reports.
  • Service chính: WasteReportServiceImpl.
  • Repository: WasteReportRepository với một số query thống kê.

Luồng create/update report:

  • Validate input (ảnh, toạ độ, category, số lượng, loại file) trong validateCreateRequest.
  • Chống spam report gần nhau (khoảng 30m, trong 10 phút) bằng query countRecentNearDuplicate.
  • Tạo bản ghi WasteReport, WasteReportItem (chi tiết loại rác).
  • Upload ảnh lên Cloudinary qua CloudinaryService, lưu URL vào WasteReport & ReportImage.
// Ví dụ mapping entity cơ bản (WasteReport)
@Entity
@Table(
    name = "waste_reports",
    indexes = {
        @Index(name = "idx_wr_citizen_status_created", columnList = "citizen_id,status,created_at"),
        @Index(name = "idx_wr_status_created", columnList = "status,created_at")
    }
)
public class WasteReport {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "citizen_id", nullable = false)
    private Citizen citizen;
    ...
}

2.2. Yêu cầu thu gom (CollectionRequest) & tác vụ của Collector

  • Entity: CollectionRequest map với bảng collection_requests.
  • Service:
    • CollectorServiceImpl: tác vụ cho nhân viên thu gom (task list, history, update status,...).
    • EnterpriseAssignmentService, TaskAutomationServiceImpl: tự động phân công & job SLA.
  • Repository: CollectionRequestRepository với nhiều projection view phục vụ dashboard và task list.

Kiến thức quan trọng:

  • Dùng enum CollectionRequestStatus và WasteReportStatus để quản lý state máy trạng thái.
  • Dùng projection interface (vd CollectorTaskView, CollectorWorkHistoryView) để tránh N+1 và giảm dữ liệu trả về.
  • Nhiều phương thức cập nhật trạng thái dùng JPQL update trả về số dòng cập nhật để chống race condition (kiểm tra trạng thái mong đợi).

2.3. Điểm thưởng & lịch sử điểm (PointTransaction)

  • Entity: PointTransaction map bảng point_transactions.
  • Repository: PointTransactionRepository với query tổng điểm, filter theo ngày, enterprise, loại giao dịch.
  • Service: WasteReportServiceImpl (lịch sử thưởng người dân), EnterpriseStatisticsServiceImpl (thống kê enterprise).

Best practice đã dùng:

  • Không filter bằng vòng lặp Java với list lớn: lịch sử điểm dùng query findByCitizenIdAndDateRange.
  • Dùng aggregate SQL (SUM, GROUP BY) cho thống kê thay vì tính toán Java.
  • Thêm index trên citizen_id,transaction_type,created_at để tối ưu query theo user + thời gian.

2.4. Khiếu nại / Feedback (Feedback)

  • Entity: Feedback map bảng feedbacks.
  • Repository: FeedbackRepository với @EntityGraph để tránh N+1 khi xem danh sách.
  • Service:
    • WasteReportServiceImpl: công dân gửi complaint.
    • EnterpriseFeedbackServiceImpl: doanh nghiệp xem & xử lý feedback.
    • AdminComplaintServiceImpl: admin xem tất cả khiếu nại.

3. Validate & rule nghiệp vụ tiêu biểu

3.1. Validate báo cáo rác (createReport / updateReport)

  • Ảnh: không được rỗng, phải là MIME type image/*, giới hạn kích thước tối đa (10MB input, nén xuống < 1MB trước khi upload).
  • Danh sách loại rác: không rỗng, số lượng quantities phải khớp số category, quantity >= 0.
  • Toạ độ: bắt buộc, nằm trong range hợp lệ (-90 ≤ lat ≤ 90, -180 ≤ lng ≤ 180).
  • Chống spam: nếu công dân vừa gửi report ở gần vị trí đó (±0.00030 độ, ~30m) trong vòng 10 phút thì reject.
  • Update/hủy report: chỉ cho phép khi status == PENDING và chưa có CollectionRequest liên quan.

3.2. Rule SLAs & tự động hoá

  • Job checkAssignedTasksTimeout: nếu collector không nhận task trong X giờ:
    • Tăng số lần vi phạm của collector.
    • Huỷ phân công, trả task về trạng thái enterprise (ACCEPTED_ENTERPRISE).
    • Ghi log vào CollectionTracking.
  • Job checkSlaViolations: nếu thời gian xử lý vượt SLA:
    • Đánh dấu slaViolated = true cho task.
    • Tăng violation count collector và có thể chuyển collector sang trạng thái SUSPEND khi quá ngưỡng.

3.3. Rule khiếu nại (complaints/feedback)

  • Chỉ cho phép các loại complaint chuẩn hoá: COMPLAINT_COLLECTION, COMPLAINT_REWARD, COMPLAINT_SYSTEM.
  • Khiếu nại về thưởng (COMPLAINT_REWARD) chỉ hợp lệ nếu báo cáo đã ở trạng thái COLLECTED.
  • Enterprise/Admin giải quyết complaint:
    • Cập nhật status và resolution của feedback.
    • Nếu xác nhận complaint đúng (status = RESOLVED) thì tăng violation cho collector.

4. Bảo mật (Security)

4.1. Xác thực & uỷ quyền (AuthZ/AuthN)

  • JWT:
    • Client đăng nhập (qua AuthServiceImpl) nhận về access token (JWT) + có thể refresh token.
    • JWT chứa claim sub (email), roles, jti (ID duy nhất của token), exp (hết hạn).
  • Spring Security:
    • Cấu hình trong SecurityConfig:
      • Filter JWT đọc token từ header Authorization: Bearer ....
      • Phân quyền dựa trên role (ADMIN / ENTERPRISE / COLLECTOR / CITIZEN).
  • Token denylist:
    • Bảng invalidated_tokens + service TokenDenylistService dùng để thu hồi token khi logout/ban.
    • Check mỗi request: nếu jti tồn tại trong denylist và chưa hết hạn → token bị từ chối.

4.2. Bảo vệ dữ liệu & input

  • Validate mạnh ở tầng service:
    • Không tin dữ liệu từ client: luôn check trạng thái, quyền sở hữu, loại dữ liệu.
    • Ví dụ: chỉ owner report mới được update/hủy; enterprise chỉ xem được report trong khu vực/service area của mình.
  • Tránh lộ dữ liệu nhạy cảm trong log:
    • CollectionRequest dùng @ToString(exclude = { "report", "enterprise", "collector" }) để log không in tràn nội dung liên quan.
  • Upload file:
    • Chỉ cho phép các extension ảnh: jpg, jpeg, png, gif, bmp (check bằng FileUpLoadUtil.assertAllowedImage).
    • Giới hạn kích thước file, kiểm tra MIME type bắt đầu bằng image/.

4.3. Một số lưu ý bảo mật khác

  • Không trả về thông tin nội bộ khi lỗi: dùng AppException và ErrorCode để chuẩn hoá message.
  • Kiểm tra chặt chẽ quyền truy cập:
    • Enterprise chỉ xem/giải quyết feedback thuộc enterprise của mình (check feedback.collectionRequest.enterprise.id).
    • Collector chỉ xem task thuộc về mình (findByIdAndCollector_Id hoặc kiểm tra lại trong handleUpdateError).

5. API tiêu biểu & luồng nghiệp vụ

5.1. Luồng báo cáo rác của công dân

  1. Công dân tạo report
    • API: POST /api/citizen/reports (request: CreateWasteReportRequest).
    • Service: WasteReportServiceImpl.createReport.
    • Bước:
      1. Lấy citizen từ email trong JWT (requireCitizenByEmail).
      2. Validate input (ảnh, category, quantity, toạ độ) bằng validateCreateRequest.
      3. Check spam bằng wasteReportRepository.countRecentNearDuplicate.
      4. Tạo WasteReport + WasteReportItem.
      5. Upload ảnh → Cloudinary → lưu link.
      6. Trả về WasteReportResponse với thông tin cơ bản + ảnh + category.
  2. Enterprise xem & duyệt report
    • API: GET /api/enterprise/reports (filter theo status).
    • Service: EnterpriseWasteReportServiceImpl.getReports.
    • Bước:
      1. Xác thực enterprise (id từ JWT hoặc param) qua validateEnterprise.
      2. Lấy list report theo trạng thái, sort theo createdAt.
      3. Lọc theo khu vực mà enterprise phục vụ (dùng AddressMatchUtil.isInServiceArea).
      4. Load thêm ảnh, items, liên kết với CollectionRequest để build EnterpriseWasteReportResponse.
  3. Enterprise chấp nhận / từ chối report
    • API: POST /api/enterprise/reports/{id}/accept, /reject.
    • Service: EnterpriseWasteReportServiceImpl.acceptReport / rejectReport.

5.2. Luồng nhiệm vụ của Collector

  1. Lấy danh sách task
    • API: GET /api/collector/tasks (có thể filter theo status / all).
    • Service: CollectorServiceImpl.getTasks.
    • Repository: CollectionRequestRepository.findTasksForCollector... trả về projection CollectorTaskView.
  2. Collector nhận task / bắt đầu / hoàn thành / từ chối
    • API:
      • POST /api/collector/tasks/{id}/accept
      • POST /api/collector/tasks/{id}/start
      • POST /api/collector/tasks/{id}/complete
      • POST /api/collector/tasks/{id}/reject
    • Service: CollectorServiceImpl.acceptTask, startTask, completeTask, rejectTask.
    • Rule:
      • Chỉ đổi trạng thái nếu trạng thái hiện tại đúng mong đợi (ASSIGNED → ACCEPTED_COLLECTOR → ON_THE_WAY → COLLECTED).
      • Kiểm tra quyền sở hữu task (task phải thuộc collector đang đăng nhập).
      • Update đồng thời CollectionRequest và WasteReport.status, ghi log vào CollectionTracking.
  3. Lịch sử làm việc & thống kê
    • API: GET /api/collector/work-history, /stats, /waste-volume...
    • Service: CollectorServiceImpl.getWorkHistory, getStats, getWasteVolumeStats, getWasteTypeStats.
    • Tất cả thống kê sử dụng aggregate SQL (COUNT, SUM, GROUP BY) qua CollectionRequestRepository, CollectorReportItemRepository.

Chi tiết endpoint Collector & validate

  • GET /api/collector/dashboard
    • Controller: CollectorController.getDashboard
    • Service/Repo: dùng CollectorJwtSupport.extractCollectorId lấy collectorId, sau đó CollectorRepository.findById để lấy status.
    • Validate:
      • Bắt buộc role COLLECTOR (@PreAuthorize("hasRole('COLLECTOR')")).
      • Nếu không tìm thấy collector theo id trong JWT → trả về status = null, online = false (không ném exception để dashboard vẫn hiển thị được).
  • GET /api/collector/stats/performance
    • Controller: CollectorController.getPerformanceStats
    • Service: CollectorServiceImpl.getStats
    • Validate & luồng:
      • Lấy collectorId từ JWT, nếu không hợp lệ thì Spring Security chặn trước.
      • Nếu không truyền year → mặc định lấy năm hiện tại.
      • Dùng CollectionRequestRepository.countByCollector_IdAndStatus đếm số task COMPLETED và COLLECTED.
      • Dùng query countCompletedByMonth để lấy thống kê theo từng tháng → map sang DTO CollectorMonthlyCompletedCountResponse.
  • GET /api/collector/stats/waste-volume
    • Controller: CollectorController.getWasteVolumeStats
    • Service: CollectorServiceImpl.getWasteVolumeStats
    • Validate & luồng:
      • Nếu không truyền year → lấy năm hiện tại.
      • Validate: 1970 ≤ year ≤ 2100, nếu sai → 400 BAD_REQUEST với message "Năm không hợp lệ (phải từ 1970 đến 2100)".
      • Tính khoảng thời gian [01/01/year, 01/01/year+1) làm filter.
      • Dùng query sumCompletedWeightByMonthForCollector, sumCompletedWeightByQuarterForCollector để tổng hợp totalWeightKg, totalRequests, xử lý null → 0.
  • GET /api/collector/stats/waste-type
    • Controller: CollectorController.getWasteTypeStats
    • Service: CollectorServiceImpl.getWasteTypeStats
    • Luồng:
      • Dùng repo CollectorReportItemRepository.sumWeightByWasteTypeForCollector trả về list Object[] (wasteTypeCode, totalWeight).
      • Map thành Map<String, BigDecimal>, nếu weight == null → thay bằng BigDecimal.ZERO.
  • GET /api/collector/stats/tasks
    • Controller: CollectorController.getTaskStatusCounts
    • Service: CollectorServiceImpl.getTaskStatusCounts
    • Validate & rule:
      • Repo trả về list projection CollectorTaskStatusCountView (status, total).
      • Map về Map<status, total>, xử lý null → 0.
      • Luôn trả đủ tất cả state trong enum CollectionRequestStatus (kể cả những trạng thái chưa có bản ghi nào, count = 0) để UI render cố định.
  • GET /api/collector/stats/general
    • Controller: CollectorController.getGeneralStats
    • Service: CollectorServiceImpl.getGeneralStats
    • Luồng:
      • Nhận bộ lọc day/month/year, truyền thẳng xuống CollectionRequestRepository.getStatsByDate.
      • Nếu repo không trả dữ liệu → trả về DTO với totalWeight = 0, totalTasks = 0.
      • Nếu có dữ liệu: đọc row[0] (tổng khối lượng), row[1] (số task) với fallback null → 0.
  • GET /api/collector/leaderboard
    • Controller: CollectorController.getLeaderboard
    • Service: CollectorServiceImpl.getLeaderboard
    • Luồng:
      • Dùng query getLeaderboardByTaskCount(month, year) để lấy collectorId, fullName, totalTasks, totalWeight.
      • Map sang CollectorLeaderboardResponse và tự tăng rank từ 1 → n.
      • NULL totalTasks hoặc totalWeight được thay bằng giá trị 0 tương ứng.
  • GET /api/collector/history
    • Controller: CollectorController.getWorkHistory
    • Service: CollectorServiceImpl.getWorkHistory
    • Validate & luồng:
      • Cho phép filter theo status (string); nếu rỗng/null → gọi findWorkHistoryForCollector (tất cả lịch sử).
      • Nếu có status → gọi findWorkHistoryForCollectorByStatus.
      • Dùng projection CollectorWorkHistoryView để trả về các field cần cho UI (requestCode, enterpriseName, wasteTypeName, thời gian,...).
  • PATCH /api/collector/status
    • Controller: CollectorController.updateMyStatus (body: UpdateCollectorStatusRequest).
    • Service: CollectorServiceImpl.updateAvailabilityStatus
    • Rule validate:
      • Truyền status dạng string (AVAILABLE/ACTIVE/INACTIVE/SUSPEND).
      • Nếu null/rỗng → 400 "Trạng thái không được để trống".
      • Nếu không map được sang enum CollectorStatus → 400 "Trạng thái không hợp lệ".
      • Không cho phép tự chuyển sang SUSPEND → 403.
      • Nếu không tìm thấy collector theo collectorId → 404.
  • PUT /api/collector/profile
    • Controller: CollectorController.updateMyProfile
    • Service: ProfileService.updateCollectorProfile
    • Validate (tầng service):
      • Check collector tồn tại theo collectorId lấy từ JWT.
      • Validate các trường cơ bản: tên, email, phương tiện, biển số, số điện thoại (format, độ dài,...).
      • Đảm bảo không vi phạm unique email/phone nếu có constraint ở DB.

5.3. Luồng điểm thưởng & leaderboard công dân

  1. Lịch sử điểm của công dân
    • API: GET /api/citizen/points/history (có thể filter theo khoảng thời gian).
    • Service: WasteReportServiceImpl.getRewardHistory.
    • Validate:
      • Lấy citizen từ email, nếu không tồn tại → ErrorCode.USER_NOT_EXISTED.
      • Nếu có startDate/endDate: dùng query filter theo createdAt để không load toàn bộ lịch sử.
  2. Tổng điểm & thống kê theo tháng/quý/năm
    • API: GET /api/citizen/points/summary.
    • Service: WasteReportServiceImpl.getMyPointSummary.
    • Rule:
      • Validate year, quarter, month trong range hợp lệ, nếu sai trả HTTP 400.
      • Dùng query sumPointsByCitizenIdAndTypeAndRange với điều kiện thời gian đã chuẩn hoá.
  3. Leaderboard công dân
    • API: GET /api/citizen/leaderboard (filter theo ngày/tháng/năm hoặc khu vực).
    • Repo: PointTransactionRepository.findCitizenLeaderboard (GROUP BY, SUM(points)).

5.x. Bảng tổng hợp Endpoint toàn hệ thống

Mục này liệt kê toàn bộ endpoint REST trong hệ thống (theo từng module), kèm: HTTP method, URL, Controller method, Service chính và các điểm validate / rule nghiệp vụ quan trọng (ở mức tóm tắt, chi tiết logic xem trong service tương ứng).

5.x.1. Auth

  • POST /api/auth/introspect → AuthController.introspect → AuthService.introspect
    • Kiểm tra token có hợp lệ, chưa hết hạn, chưa bị revoke trong denylist (bảng invalidated_tokens + cache).
  • POST /api/auth/register → AuthController.register → AuthService.register
    • Validate email không trùng, mật khẩu đủ mạnh, role hợp lệ; tạo user + thực thể domain (citizen/collector/enterprise) tương ứng.
  • POST /api/auth/login → AuthController.login → AuthService.login
    • Check credential, trạng thái tài khoản (active/suspended), trả về JWT với roles, jti, exp.
  • POST /api/auth/logout → AuthController.logout → AuthService.logout
    • Lấy token từ header Authorization hoặc body, parse JWT, ghi jti vào denylist đến khi hết hạn.

5.x.2. Citizen

  • GET /api/citizen/reports/stats → CitizenController.getMyReportStats → WasteReportService.getMyReportStats
  • POST /api/citizen/reports (multipart) → createReport → WasteReportService.createReport
    • Validate ảnh, toạ độ, danh sách loại rác, chống spam như mô tả ở mục 3.1.
  • PUT /api/citizen/reports/{id} (multipart) → updateReport → WasteReportService.updateReport
    • Chỉ cho phép owner sửa khi report đang PENDING và chưa có CollectionRequest.
  • DELETE /api/citizen/reports/{id} → deleteReport → WasteReportService.deleteReport
  • GET /api/citizen/reports → getMyReports → WasteReportService.getMyReports
  • GET /api/citizen/reports/{id} → getMyReportById → WasteReportService.getMyReportById
  • GET /api/citizen/reports/{id}/result → getMyReportResult → WasteReportService.getMyReportResult
  • GET /api/citizen/rewards/history → getRewardHistory → WasteReportService.getRewardHistory
    • Validate khoảng thời gian; nếu có startDate/endDate dùng filter DB, không load toàn bộ history.
  • GET /api/citizen/leaderboard → getLeaderboard → WasteReportService.getLeaderboard
  • GET /api/citizen/points/summary → getMyPointSummary → WasteReportService.getMyPointSummary
  • POST /api/citizen/complaints (multipart) → createComplaint → WasteReportService.createComplaint
  • GET /api/citizen/complaints → getComplaints → WasteReportService.getComplaints

6. Tối ưu hiệu năng & tránh N+1

6.1. Nén & upload ảnh

  • Nén ảnh: FileUpLoadUtil.compressImage
    • Nếu file <= 1MB: giữ nguyên bytes.
    • Nếu lớn hơn: đọc vào BufferedImage, resize 1 lần về tối đa 1920x1080 rồi thử nén với một vài mức quality (0.85 → 0.7 → 0.55 → 0.4).
    • Nếu vẫn >1MB: fallback resize mạnh hơn về 800x800, nén ~0.6.
  • Upload ảnh Cloudinary: WasteReportServiceImpl.saveReportImages
    • Dùng ExecutorService + CompletableFuture để upload song song (tối đa 4 ảnh cùng lúc).
    • Ảnh đầu tiên vẫn là ảnh đại diện; tất cả ảnh lưu vào bảng ReportImage.

6.2. Tránh N+1 bằng EntityGraph & projection

  • WasteReportRepository: findAll(Sort), findByStatus(...) dùng @EntityGraph("citizen","citizen.user") để không N+1 khi lấy tên/email người báo cáo.
  • FeedbackRepository: findAll, findByEnterpriseId fetch sẵn citizen, citizen.user, collectionRequest, collectionRequest.collector, collectionRequest.enterprise.
  • PointTransactionRepository: findByCitizenId, findByCitizenIdAndDateRange fetch sẵn citizen để mapper dùng thông tin công dân không sinh thêm query.
  • CollectionRequestRepository: nhiều query trả về projection interface (view) thay vì entity, giúp tránh lazy-loading & N+1 cho task list / history / leaderboard.

6.3. Index ở cấp DB

  • waste_reports: index theo citizen_id,status,created_at, status,created_at.
  • collection_requests: index theo collector_id,status, enterprise_id,status,created_at, status,assigned_at, sla_violated,started_at, report_id.
  • point_transactions: index theo citizen_id,created_at, citizen_id,transaction_type,created_at.
  • invalidated_tokens: index theo expiryTime hỗ trợ dọn dẹp.
  • feedbacks: index theo citizen_id,created_at, status,created_at.
  • vouchers: index theo active,id, title.

7. Auth, Token Denylist & Redis trong dự án

  • Bảng invalidated_tokens lưu id (jti) + expiryTime cho token bị thu hồi.
  • Service TokenDenylistService:
    • Dùng ConcurrentHashMap làm cache in-memory (max 50k entry) với TTL dương/âm để giảm hit DB.
    • Khi logout hoặc revoke: ghi vào DB + update cache.
    • Khi check: ưu tiên lấy từ cache, miss thì hỏi DB rồi update cache.

7.1. Redis – Cách dùng trong dự án

  • Cấu hình:
    • Dùng starter spring-boot-starter-data-redis (Spring Boot 3.4.1) + client Lettuce.
    • Class RedisConfig tạo LettuceConnectionFactory với RedisStandaloneConfiguration (host, port, password).
    • Cấu hình RedisCacheManager làm CacheManager mặc định cho Spring Cache, TTL mặc định ~5 phút.
    • Key cache là String, value serialize bằng GenericJackson2JsonRedisSerializer.
  • Cache dashboard & thống kê Collector (tối ưu performance, giảm truy vấn MySQL):
    • Các hàm đọc nhiều dữ liệu/tổng hợp trong CollectorServiceImpl được gắn @Cacheable:
      • getTasks → cache theo collectorId, status, cờ all (danh sách nhiệm vụ).
      • getTaskDetail → cache chi tiết 1 task theo collectorId + requestId.
      • getTaskStatusCounts → cache map số lượng task theo trạng thái cho 1 collector.
      • getWorkHistory → cache lịch sử làm việc theo collectorId + status.
      • getStats, getWasteVolumeStats, getWasteTypeStats → cache dữ liệu thống kê theo năm/collector.
      • getGeneralStats → cache tổng khối lượng + số task theo bộ lọc day/month/year.
      • getLeaderboard → cache bảng xếp hạng collector theo month/year.
    • Nhờ đó:
      • Lần gọi API đầu tiên sẽ query DB và đẩy kết quả vào Redis.
      • Các lần gọi sau với cùng tham số sẽ đọc trực tiếp từ Redis (RAM) → nhanh hơn, giảm tải MySQL.
  • Invalid cache khi dữ liệu Collector thay đổi:
    • Các hàm thay đổi trạng thái nhiệm vụ hoặc trạng thái làm việc Collector trong CollectorServiceImpl được gắn @CacheEvict(allEntries = true):
      • updateStatus, acceptTask, startTask, rejectTask, completeTask, updateAvailabilityStatus.
    • Cách triển khai đơn giản cho sinh viên:
      • Mỗi khi có ghi/đổi trạng thái, xóa toàn bộ cache của các vùng: collectorTasks, collectorTaskDetail, collectorTaskStatusCounts, collectorWorkHistory, collectorPerformanceStats, collectorWasteVolumeStats, collectorWasteTypeStats, collectorGeneralStats, collectorLeaderboard.
      • Không đổi logic nghiệp vụ, chỉ thêm annotation cache ở tầng service.
  • Lưu token trong Redis (mở rộng từ denylist trong DB):
    • Entity RedisToken dùng @RedisHash("redis_token"), gồm:
      • id (string) – thường là userId/tokenId.
      • accessToken, refreshToken.
      • ttl với @TimeToLive để Redis tự xóa sau khi hết hạn.
    • Repository RedisTokenRepository (extends CrudRepository<RedisToken, String>) + service RedisTokenService cho phép:
      • save: lưu token tạm vào Redis.
      • findById: lấy token theo id.
      • deleteById: xóa khi logout/thu hồi.
    • Ý nghĩa: có thể dùng Redis như kho token/session tập trung, dễ scale nhiều instance và hỗ trợ force logout, blacklist token.
  • Lưu ý về hiệu năng & RAM khi dùng Redis:
    • Redis lưu dữ liệu trên RAM → cache càng nhiều, tốn RAM hơn nhưng đổi lại đọc rất nhanh.
    • TTL ~5 phút trong RedisConfig giúp dữ liệu cache tự hết hạn, tránh phình RAM lâu dài.
    • Các API dashboard/thống kê được cache là dữ liệu JSON vừa phải, phù hợp cho môi trường sinh viên/lab.
  • Cách chạy Redis bằng Docker & tự kiểm tra dữ liệu (dành cho sinh viên mới):
    • Bật Redis:
      • Ở thư mục gốc dự án (chứa docker-compose.yml): chạy docker-compose up -d.
      • Lệnh này khởi động container redis-local dùng image redis:7-alpine và giới hạn bộ nhớ với --maxmemory 256mb, --maxmemory-policy allkeys-lru.
      • Dùng docker ps để kiểm tra container redis-local đang chạy.
    • Xem log Redis:
      • docker logs redis-local → xem log khởi động.
      • docker logs -f redis-local → theo dõi log realtime (Ctrl+C để thoát).
    • Mở Redis CLI bên trong container:
      • docker exec -it redis-local redis-cli → vào shell Redis với prompt 127.0.0.1:6379>.
      • KEYS * → liệt kê nhanh tất cả key (chỉ nên dùng trong môi trường lab, dữ liệu ít).
      • SCAN 0 → cách an toàn hơn để duyệt key theo từng trang.
      • GET "collectorPerformanceStats:1:2025" (ví dụ) → xem dữ liệu cache JSON cho key cụ thể.
      • TTL "collectorPerformanceStats:1:2025" → xem còn bao nhiêu giây trước khi key hết hạn.
      • Gõ exit để thoát redis-cli.
    • Kịch bản test đơn giản để “thấy Redis hoạt động”:
      • B1: Khởi động Redis: docker-compose up -d.
      • B2: Gọi 1 API có cache, ví dụ GET /api/collector/stats/performance khi đang đăng nhập bằng collector.
      • B3: Mở redis-cli và chạy KEYS * → quan sát các key dạng collectorPerformanceStats:<collectorId>:<year>, collectorTasks:....
      • B4: Gọi lại cùng API thêm vài lần → kết quả vẫn đúng nhưng số query đến MySQL giảm (có thể bật spring.jpa.show-sql=true để quan sát).

7.x. Ghi chú ôn tập theo nhóm Endpoint

Phần này không liệt kê lại toàn bộ URL (đã có ở 5.x) mà tập trung vào kiến thức, nghiệp vụ và validate chính cho từng nhóm endpoint để bạn dễ ôn thi/ôn tập.

7.x.1. Nhóm Auth

  • Đăng ký (POST /api/auth/register):
    • Nghiệp vụ: tạo user + entity domain (Citizen/Collector/Enterprise) và trả JWT ngay sau đăng ký.
    • Validate: email unique & đúng format, mật khẩu đủ mạnh, role hợp lệ.
  • Đăng nhập (POST /api/auth/login):
    • Nghiệp vụ: kiểm tra credential, trả JWT chứa sub, roles, jti, exp.
    • Validate: tài khoản tồn tại, không bị SUSPEND/INACTIVE, mật khẩu khớp hash.
  • Introspect (POST /api/auth/introspect):
    • Nghiệp vụ: kiểm tra token có hợp lệ và chưa bị thu hồi.
    • Validate: verify chữ ký JWT, thời hạn, và check jti không nằm trong bảng invalidated_tokens.
  • Logout (POST /api/auth/logout):
    • Nghiệp vụ: thu hồi token hiện tại bằng cách thêm jti vào denylist đến khi hết hạn.
    • Validate: lấy token từ header/body, parse được; nếu token rỗng hoặc không parse được → lỗi 400.

7.x.2. Nhóm Citizen

  • Báo cáo rác (/api/citizen/reports):
    • Nghiệp vụ: citizen tạo/sửa/xoá report, upload ảnh, gửi vị trí, loại rác.
    • Validate:
      • Ảnh: bắt buộc, MIME image/*, dung lượng input ≤ 10MB, nén xuống < 1MB trước upload.
      • Danh sách loại rác: không rỗng, số lượng record khớp số category, quantity ≥ 0.
      • Toạ độ: lat ∈ [-90, 90], lng ∈ [-180, 180].
      • Chống spam: không cho phép report trùng vị trí (~30m) trong vòng 10 phút.
      • Update/delete: chỉ khi report thuộc về citizen hiện tại, status PENDING và chưa có CollectionRequest.
  • Điểm thưởng & leaderboard (/rewards/history, /points/summary, /leaderboard):
    • Nghiệp vụ: xem lịch sử điểm, tổng kết theo tháng/quý/năm, bảng xếp hạng citizen.
    • Validate: kiểm tra tồn tại citizen; tham số thời gian (year/quarter/month, startDate/endDate) trong range hợp lệ; luôn filter trên DB (không load toàn bộ rồi lọc).
  • Khiếu nại (/complaints):
    • Nghiệp vụ: citizen gửi complaint về thu gom/điểm thưởng/hệ thống.
    • Validate: loại complaint nằm trong tập cho phép; với complaint về điểm thưởng thì report phải đã COLLECTED; chỉ được complaint trên report của chính mình.
  • Hồ sơ & danh mục (/profile, /waste-categories):
    • Nghiệp vụ: cập nhật thông tin cá nhân; lấy danh sách loại rác.
    • Validate: format/độ dài các trường (email, phone, address); không vi phạm unique (nếu có) trên DB.

7.x.3. Nhóm Collector

  • Task & trạng thái (/api/collector/collections/...):
    • Nghiệp vụ: xem task list, accept/start/reject/collected, tạo report thu gom.
    • Validate:
      • Mọi thao tác luôn check task thuộc về collector hiện tại (findByIdAndCollector_Id hoặc kiểm tra sau khi query).
      • Cập nhật trạng thái dùng JPQL update có điều kiện trạng thái cũ (ASSIGNED → ACCEPTED_COLLECTOR → ON_THE_WAY → COLLECTED), nếu mismatch → lỗi 400.
      • Khi reject phải có lý do; chỉ reject khi status là ASSIGNED.
      • Mỗi lần đổi trạng thái đều cập nhật WasteReport.status tương ứng và ghi log CollectionTracking.
  • Dashboard & thống kê (/api/collector/stats/..., /leaderboard, /history):
    • Nghiệp vụ: xem hiệu suất, khối lượng rác theo tháng/quý, thống kê theo loại rác, trạng thái task, leaderboard.
    • Kiến thức:
      • Dùng aggregate SQL (COUNT, SUM, GROUP BY) + projection interface để tránh N+1.
      • Luôn trả đủ mọi trạng thái trong enum CollectionRequestStatus với count=0 nếu không có dữ liệu (giúp UI ổn định).
      • Validate năm trong khoảng [1970, 2100].
  • Trạng thái làm việc & hồ sơ (/api/collector/status, /profile):
    • Nghiệp vụ: collector tự chuyển trạng thái (ONLINE/OFFLINE/AVAILABLE/ACTIVE/INACTIVE) và cập nhật thông tin cá nhân.
    • Validate: không cho phép tự chuyển sang SUSPEND; status string phải map được sang enum; kiểm tra tồn tại collector trước khi update.
  • Feedback (/api/collector/feedbacks):
    • Nghiệp vụ: collector gửi feedback (có thể gắn với collection request) và xem lại các feedback của mình.
    • Validate: feedback phải gắn đúng với collector hiện tại; nếu có collectionRequestId thì request đó cũng phải thuộc collector.

7.x.4. Nhóm Enterprise

  • Accept/Reject & phân công (/api/enterprise/requests/...):
    • Nghiệp vụ: enterprise nhận report từ citizen, quyết định accept/reject, tạo request và gán collector phù hợp.
    • Validate: enterprise chỉ thao tác trên report trong phạm vi của mình; chỉ accept report PENDING; chỉ gán collector ở trạng thái phù hợp và trong bán kính cho phép.
  • Xem report & collector report (/waste-reports, /collector-reports):
    • Nghiệp vụ: xem danh sách/chi tiết báo cáo rác và báo cáo thu gom của các collector thuộc mình.
    • Validate: mọi query đều filter theo enterpriseId lấy từ JWT; dùng @EntityGraph hoặc projection để tối ưu.
  • Feedback (/enterprise/feedbacks):
    • Nghiệp vụ: enterprise xem và giải quyết complaint từ citizen, đồng thời đánh giá collector.
    • Validate: chỉ feedback gắn với enterprise hiện tại mới được truy cập; khi resolve có thể tăng violation count cho collector nếu complaint đúng.
  • Statistics & voucher (/enterprise/reports/..., /enterprise/vouchers):
    • Nghiệp vụ: xem thống kê khối lượng rác, báo cáo theo ngày/tháng, điểm citizen; quản lý voucher.
    • Validate: tham số thời gian hợp lệ; với voucher chỉ enterprise owner (hoặc ENTERPRISE_ADMIN) mới được CRUD.

7.x.5. Nhóm Admin

  • Accounts & quyền (/api/admin/accounts):
    • Nghiệp vụ: admin tạo account cho mọi role, khoá/mở khoá tài khoản, xem danh sách và chi tiết user.
    • Validate: không cho phép admin tự khoá chính mình; mọi hành động đều gắn với adminEmail lấy từ JWT.
  • Complaints & reports (/api/admin/complaints, /api/admin/reports/...):
    • Nghiệp vụ: admin giám sát toàn bộ complaint và report (citizen + collector), có thể đưa ra quyết định cuối cùng.
    • Kiến thức: dùng lại service enterprise cho phần lớn truy vấn nhưng bỏ giới hạn theo enterpriseId (hoặc cho phép truyền enterpriseId làm filter tuỳ chọn).
  • Waste category & point rule (/api/admin/waste-categories, /api/admin/point-rule):
    • Nghiệp vụ: cấu hình loại rác, đơn vị, điểm thưởng/đơn vị; quy tắc điểm tổng thể.
    • Validate: ensure các giá trị (điểm, hệ số) không âm và hợp lý; update kiểu PATCH chỉ thay đổi trường được gửi lên.
  • Analytics (/api/admin/analytics/...):
    • Nghiệp vụ: dashboard admin cho toàn hệ thống (khối lượng rác, leaderboard, thống kê theo đơn vị,...).
    • Kiến thức: tất cả các endpoint này là read-only, dùng aggregate SQL với index để đảm bảo hiệu năng.

8. Hướng phát triển / tối ưu tiếp

  • Bổ sung phân trang cho các API hiện đang trả full list (complaints, feedbacks, voucher list, một số task list).
  • Có thể thêm Redis cho:
    • Cache denylist JWT (thay local cache, chia sẻ giữa nhiều instance).
    • Cache thống kê dashboard (enterprise, collector, citizen leaderboard) với TTL ngắn.
  • Bổ sung test cho các rule quan trọng: chống spam report, SLA, complaint/resolution.
