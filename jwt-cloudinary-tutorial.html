Tutorial JWT & Cloudinary + cách cấu hình trong dự án

Mục tiêu: hiểu JWT hoạt động thế nào trong Spring Security của dự án, Cloudinary được dùng để upload/xoá ảnh ra sao, và cần set cấu hình (env vars / application.yml) như thế nào để chạy được.

Lưu ý bảo mật: Secret/credentials không nên hardcode trong repo. Nên đặt bằng biến môi trường và xoay vòng (rotate) nếu đã từng lộ.

1) JWT là gì (nhìn nhanh)

  • JWT Chuỗi token gồm 3 phần: header.payload.signature. Payload có thể decode được, signature dùng để chống sửa.
  • Stateless Server không cần lưu session; mỗi request client gửi Authorization: Bearer <token>.
  • AuthZ Phân quyền dựa trên role/permission trong claims (ở dự án này: claim scope).
  • Expiry Token nên có exp ngắn để giảm rủi ro bị đánh cắp.

2) JWT đang được implement trong dự án như thế nào?

2.1. Các thành phần chính (đúng theo code hiện tại)

  • Phát hành token: JWTHelper (HS512, tạo jti, exp, sub, scope).
  • Login/Register/Logout/Introspect: AuthController gọi AuthServiceImpl.
  • Verify token cho API: Spring Security Resource Server + NimbusJwtDecoder trong SecurityConfig.
  • Thu hồi token: denylist theo jti (DB bảng invalidated_tokens + cache) qua TokenDenylistService.

2.2. Luồng chạy thực tế

  1. Client gọi login tới POST /api/auth/login (body: email/password).
  2. Backend tạo JWT trong AuthServiceImpl bằng JWTHelper.issueToken(...).
  3. Client gọi API khác kèm header Authorization: Bearer ....
  4. Spring Security verify bằng JwtDecoder (HS512) và validator timestamp + denylist.
  5. Map quyền: claim scope là chuỗi các quyền cách nhau bằng khoảng trắng (vd ROLE_ADMIN PERM_X), được map thành authorities.

2.3. Claims quan trọng trong token của dự án

Trong JWTHelper, token có những claim tiêu biểu sau:

  • sub: email user
  • iss: issuer (team2.com)
  • iat, exp: thời gian phát hành/hết hạn
  • jti: UUID để thu hồi token theo denylist
  • scope: danh sách quyền (role + permission codes) cách nhau bằng khoảng trắng
  • role, citizenId, collectorId, enterpriseId: hỗ trợ nghiệp vụ

2.4. Logout và denylist (thu hồi token) hoạt động ra sao?

  • Logout POST /api/auth/logout nhận token từ header hoặc body, rồi lưu jti vào bảng invalidated_tokens đến khi token hết hạn.
  • Validate Mọi request sau đó sẽ bị reject nếu token có jti nằm trong denylist.
  • Lợi ích JWT vốn stateless; denylist giúp “revoke” token đã phát hành.

3) Cấu hình JWT trong dự án (application.yml + env vars)

Dự án đọc cấu hình ở src/main/resources/application.yml và ưu tiên biến môi trường.

  • jwt.signerKey → biến môi trường JWT_SIGNER_KEY
  • jwt.valid-duration (giây) → biến môi trường JWT_VALID_DURATION
  • app.cors.allowed-origins → biến môi trường APP_CORS_ALLOWED_ORIGINS (comma-separated)
Khuyến nghị: Với HS512, hãy dùng secret đủ dài (tối thiểu 64 bytes ngẫu nhiên) và chỉ set qua env vars khi chạy local/CI/prod.

3.1. Ví dụ set biến môi trường khi chạy local

# PowerShell (Windows)
$env:JWT_SIGNER_KEY="YOUR_LONG_RANDOM_SECRET"
$env:JWT_VALID_DURATION="3600"
$env:APP_CORS_ALLOWED_ORIGINS="http://localhost:5173, http://localhost:3000"

3.2. Request mẫu dùng JWT

# Login (nhận token trong ApiResponse.result.token hoặc field tương đương)
curl -X POST http://localhost:8080/api/auth/login ^
  -H "Content-Type: application/json" ^
  -d "{\"email\":\"demo@example.com\",\"password\":\"123456\"}"

# Gọi API cần đăng nhập
curl -X GET http://localhost:8080/api/citizen/profile ^
  -H "Authorization: Bearer <ACCESS_TOKEN>"

4) Cloudinary là gì và dự án dùng như thế nào?

  • Cloudinary Dịch vụ lưu trữ ảnh/video; trả về URL (secure_url) và public_id.
  • Trong dự án Cloudinary chỉ dùng cho ảnh (resource_type=image), có nén ảnh trước khi upload.

4.1. Các thành phần chính trong dự án

  • Bean Cloudinary: CloudinaryConfig tạo Cloudinary từ cloudinary.cloud-name/api-key/api-secret.
  • Service upload/xoá: CloudinaryServiceImpl (validate module, nén ảnh, upload, trả publicId và url).
  • API upload/xoá ảnh: FileController dưới path /api/files.

4.2. Endpoint upload/xoá ảnh

  • POST /api/files/images (multipart): file + module (mặc định: reports).
  • DELETE /api/files/images?publicId=... để xoá.

4.3. module được hỗ trợ (mapping folder)

Tham số module sẽ được normalize để map vào folder. Các module hợp lệ:

  • reports
  • requests
  • feedbacks
  • collectorReport
  • vouchers

5) Cấu hình Cloudinary trong dự án (application.yml + env vars)

  • cloudinary.cloud-name → CLOUDINARY_CLOUD_NAME
  • cloudinary.api-key → CLOUDINARY_API_KEY
  • cloudinary.api-secret → CLOUDINARY_API_SECRET
  • cloudinary.folder (tuỳ chọn) → prefix folder mặc định (vd swp)

5.1. Ví dụ set biến môi trường cho Cloudinary

# PowerShell (Windows)
$env:CLOUDINARY_CLOUD_NAME="your_cloud_name"
$env:CLOUDINARY_API_KEY="your_api_key"
$env:CLOUDINARY_API_SECRET="your_api_secret"
$env:CLOUDINARY_FOLDER="swp"

5.2. Request mẫu upload ảnh (multipart)

# Upload ảnh (cần Bearer token vì endpoint không nằm trong PUBLIC_ENDPOINTS)
curl -X POST "http://localhost:8080/api/files/images?module=reports" ^
  -H "Authorization: Bearer <ACCESS_TOKEN>" ^
  -F "file=@D:\path\to\image.jpg"

6) Deploy bằng Docker + MySQL (đúng theo cấu hình dự án)

Backend là Spring Boot (Java 21) và có sẵn Dockerfile build jar bằng Maven. Datasource MySQL được cấu hình bằng biến môi trường (có nhiều alias: MYSQL_HOST/MYSQLHOST, MYSQL_PORT/MYSQLPORT, ...).

6.1. Dockerfile đang làm gì?

  • Build stage Dùng maven:3.9.6-eclipse-temurin-21 để build jar: mvn clean package -DskipTests.
  • Run stage Dùng eclipse-temurin:21-jre-jammy để chạy java -jar app.jar trên port 8080.

Dockerfile hiện tại (nguyên lý multi-stage):

FROM maven:3.9.6-eclipse-temurin-21 AS build
WORKDIR /app
COPY . .
RUN mvn clean package -DskipTests

FROM eclipse-temurin:21-jre-jammy
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","app.jar"]
  • Vì sao dùng multi-stage: image chạy (JRE) nhỏ hơn, không cần mang Maven/source code sang runtime.
  • COPY . .: copy toàn bộ repo vào stage build. Khi repo có nhiều file, build có thể chậm nếu thiếu .dockerignore.
  • mvn ... -DskipTests: build nhanh hơn nhưng bỏ test. Nếu deploy prod, nên chạy test ở CI trước khi build image.
  • COPY /target/*.jar: lấy jar build ra và rename thành app.jar để chạy thống nhất.
Gợi ý cải thiện (tuỳ team): thêm .dockerignore (bỏ target/, node_modules/, file log), và tách bước copy pom.xml trước để tận dụng cache layer khi build.

6.2. Chạy nhanh bằng Docker (dùng DB bên ngoài)

# Build image
docker build -t swp-backend .

# Run container (trỏ tới MySQL bên ngoài bằng env vars)
docker run --rm -p 8080:8080 ^
  -e SPRING_PROFILES_ACTIVE=dev ^
  -e MYSQL_HOST=host.docker.internal ^
  -e MYSQL_PORT=3306 ^
  -e MYSQL_DATABASE=swp ^
  -e MYSQL_USER=root ^
  -e MYSQL_PASSWORD=your_password ^
  -e JWT_SIGNER_KEY=YOUR_LONG_RANDOM_SECRET ^
  -e JWT_VALID_DURATION=3600 ^
  -e APP_CORS_ALLOWED_ORIGINS="http://localhost:5173" ^
  -e CLOUDINARY_CLOUD_NAME=your_cloud_name ^
  -e CLOUDINARY_API_KEY=your_api_key ^
  -e CLOUDINARY_API_SECRET=your_api_secret ^
  swp-backend

6.3. Chạy MySQL + Backend cùng nhau bằng Docker Compose

Dự án hiện chưa có sẵn docker-compose.yml. Bạn có thể tạo file compose theo mẫu dưới đây (đặt cạnh Dockerfile). 

services:
  mysql:
    image: mysql:8.4
    container_name: swp-mysql
    environment:
      MYSQL_DATABASE: swp
      MYSQL_ROOT_PASSWORD: rootpass
      TZ: Asia/Ho_Chi_Minh
    ports:
      - "3306:3306"
    volumes:
      - swp_mysql_data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-prootpass"]
      interval: 5s
      timeout: 3s
      retries: 20

  backend:
    build: .
    container_name: swp-backend
    depends_on:
      mysql:
        condition: service_healthy
    environment:
      SPRING_PROFILES_ACTIVE: dev
      MYSQL_HOST: mysql
      MYSQL_PORT: 3306
      MYSQL_DATABASE: swp
      MYSQL_USER: root
      MYSQL_PASSWORD: rootpass
      JWT_SIGNER_KEY: YOUR_LONG_RANDOM_SECRET
      JWT_VALID_DURATION: 3600
      APP_CORS_ALLOWED_ORIGINS: "http://localhost:5173"
      CLOUDINARY_CLOUD_NAME: your_cloud_name
      CLOUDINARY_API_KEY: your_api_key
      CLOUDINARY_API_SECRET: your_api_secret
    ports:
      - "8080:8080"

volumes:
  swp_mysql_data:

6.4. Kiến thức MySQL trong dự án (những điểm cần hiểu khi deploy)

  • Driver Dùng MySQL connector (runtime) và Spring Data JPA/Hibernate.
  • JDBC URL Format: jdbc:mysql://{host}:{port}/{db}?useSSL=false&allowPublicKeyRetrieval=true&serverTimezone=UTC....
  • DDL Mặc định ddl-auto=update (tự update schema khi chạy). Profile prod dùng validate.
  • Timezone JSON/JPA dùng Asia/Ho_Chi_Minh; JDBC URL đang set serverTimezone=UTC nên cần nhất quán khi xem dữ liệu timestamp.
  • Persistence Khi chạy MySQL bằng container, nên mount volume để không mất dữ liệu khi restart.

7) Checklist debug nhanh (khi JWT/Cloudinary/Docker/MySQL không chạy)

JWT

  • 401 liên tục: kiểm tra gửi đúng header Authorization: Bearer ....
  • Token “invalid”: kiểm tra JWT_SIGNER_KEY có giống môi trường tạo token không.
  • Vừa logout xong bị 401: đúng kỳ vọng vì denylist đã thu hồi jti.
  • CORS lỗi trên browser: set đúng APP_CORS_ALLOWED_ORIGINS (danh sách origin, cách nhau bằng dấu phẩy).

Cloudinary

  • 500 “Thiếu cấu hình Cloudinary”: thiếu 1 trong 3 biến CLOUDINARY_CLOUD_NAME/API_KEY/API_SECRET.
  • 400 “module không hợp lệ”: chỉ dùng các module hợp lệ đã liệt kê.
  • Upload fail: kiểm tra file là ảnh hợp lệ và không vượt giới hạn upload (Spring multipart max 10MB).

Docker/MySQL

  • Backend không connect DB: kiểm tra MYSQL_HOST (trong compose phải là mysql), port 3306, user/pass.
  • MySQL container restart liên tục: kiểm tra biến MYSQL_ROOT_PASSWORD và quyền ghi volume.
  • Lỗi schema khi dùng profile prod: do ddl-auto=validate, cần đảm bảo schema đã được tạo trước.

8) File/code cần đọc trong dự án (để trace nhanh)

  • src/main/resources/application.yml: cấu hình JWT/CORS/Cloudinary
  • CONFIGURATION.md: giải thích env vars, profile dev/prod, MySQL/JPA
  • Dockerfile: build/run container backend
  • config/SecurityConfig: cấu hình Spring Security + JwtDecoder + CORS
  • util/JWTHelper: cách tạo token và claims
  • controller/authentication/AuthController: login/register/logout/introspect
  • service/security/TokenDenylistService: revoke token theo jti
  • config/CloudinaryConfig + service/impl/CloudinaryServiceImpl + controller/cloudinary/FileController

9) Giải thích code quan trọng trong dự án (đọc theo luồng)

9.1. Spring Security + JWT validate (SecurityConfig)

  • Public endpoints: các path trong PUBLIC_ENDPOINTS được permitAll, còn lại bắt buộc authenticated.
  • Resource Server: dùng oauth2ResourceServer().jwt() để Spring tự parse/verify JWT trên header Authorization: Bearer ....
  • Thuật toán: decoder cấu hình HMAC HS512 với secret lấy từ jwt.signerKey.
  • Denylist: JwtDecoder gắn thêm validator JtiDenylistValidator để chặn token đã logout/revoke.
  • Authorities: converter đặt authorityPrefix(""), nên giá trị trong claim scope sẽ được dùng gần như “nguyên bản”.
  • CORS: đọc app.cors.allowed-origins, tách theo dấu phẩy; origin có * sẽ đi vào allowedOriginPatterns.

9.2. Cấp token và verify token (JWTHelper)

  • issueToken() gọi generateToken(), tạo JWT với sub=email, exp=now + jwt.valid-duration, jti=UUID.
  • scope được build từ role và permission codes, nối bằng khoảng trắng (vd ROLE_ADMIN PERM_A PERM_B).
  • Claims bổ sung: role, citizenId, collectorId, enterpriseId để service/controller dùng nhanh.
  • verifyToken() kiểm chữ ký HS512 + check hết hạn + check denylist theo jti.

9.3. Cơ chế revoke token (TokenDenylistService + JtiDenylistValidator)

  • invalidate(jti, exp): lưu jti vào DB (bảng invalidated_tokens) tới khi token hết hạn, đồng thời cache “revoked”.
  • isTokenRevoked(): ưu tiên đọc cache để giảm hit DB; có negative cache 10 giây cho token “chưa bị revoke”.
  • JtiDenylistValidator: Spring gọi validator này trong lúc decode token; nếu revoked → trả invalid_token.

9.4. Upload/Xoá ảnh Cloudinary (CloudinaryConfig + CloudinaryServiceImpl + FileController)

  • CloudinaryConfig: tạo bean Cloudinary từ cloudinary.cloud-name/api-key/api-secret.
  • CloudinaryServiceImpl.uploadImage(): kiểm tra credential, validate file ảnh, nén ảnh, upload bytes, trả về publicId + url.
  • Folder theo module: tham số module được normalize và map vào folder cố định (reports/requests/feedbacks/collectorReport/vouchers).
  • FileController: cung cấp API /api/files/images (POST multipart) và /api/files/images (DELETE theo publicId).
  • Auth: do endpoint upload không nằm trong PUBLIC_ENDPOINTS, khi gọi upload/xoá cần Bearer token hợp lệ.
