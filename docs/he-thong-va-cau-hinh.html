Tổng quan

Hệ thống hỗ trợ thu gom và tái chế rác dựa trên cộng đồng. Luồng chính: người dân báo cáo rác (kèm vị trí/ảnh), doanh nghiệp tiếp nhận/duyệt và phân công nhân viên thu gom, nhân viên thu gom xử lý và cập nhật trạng thái. Hệ thống có điểm thưởng và voucher để khuyến khích báo cáo đúng.

Entry point
Khởi động ứng dụng, bật scheduling và chuẩn hóa timezone UTC.
src/main/java/.../CrowdsourcedWasteCollectionRecyclingSystemApplication.java

Kiến trúc & vai trò

Vai trò (RBAC)

Role Trách nhiệm chính
ADMIN Quản trị toàn hệ thống: tài khoản, danh mục rác, khiếu nại, luật cộng điểm, phân tích số liệu.
ENTERPRISE Quản lý hoạt động thu gom: duyệt báo cáo, phân công collector, theo dõi thống kê, tạo voucher.
COLLECTOR Nhận nhiệm vụ thu gom, cập nhật tiến độ, báo cáo hoàn thành, gửi phản hồi.
CITIZEN Báo cáo rác, theo dõi trạng thái xử lý, tích điểm và đổi voucher.

Luồng nghiệp vụ mẫu (mức khái quát)

Citizen
Tạo báo cáo rác → đính kèm ảnh/vị trí → chờ Enterprise duyệt/điều phối.
Enterprise
Duyệt báo cáo → tạo yêu cầu thu gom (task) → gán Collector → giám sát SLA.
Collector
Nhận task → bắt đầu → hoàn thành → upload ảnh minh chứng → hệ thống cập nhật trạng thái.
Rewards
Cộng điểm theo luật → Citizen đổi voucher → lưu lịch sử giao dịch/voucher redemption.

Cấu trúc code

Project theo kiểu 3-layer: controller (API) → service (nghiệp vụ) → repository (DB).

Package Vai trò
config/ Các cấu hình Security/JWT/CORS/Swagger/Cloudinary/eKYC/seed/work rules.
controller/ HTTP endpoints chia theo nhóm admin/citizen/collector/enterprise/authentication.
service/, service/impl/ Nghiệp vụ: phân công, xử lý report, điểm thưởng, automation, auth.
repository/ Spring Data JPA repositories, query chuyên dụng cho từng domain.
entity/, enums/ Domain model và các trạng thái (status) của report/task/collector.
dto/ Request/Response payloads; chuẩn hóa dữ liệu đi/đến API.
exception/ AppException/ErrorCode và xử lý lỗi tập trung.

Chuẩn API & lỗi

1) Response wrapper

Phần lớn endpoint trả về cấu trúc chung ApiResponse: dto/response/ApiResponse.java. Mục tiêu là thống nhất format, dễ xử lý trên frontend. 

{
  "code": 1000,
  "message": ".... (có thể null)",
  "result": { ... }
}
code
Mã lỗi/ok của hệ thống. Mặc định 1000 nghĩa là thành công.
message
Thông báo kèm theo (khi lỗi hoặc khi muốn trả message).
result
Payload kết quả (DTO response).

2) Xử lý exception tập trung

Toàn bộ exception được gom về exception/GlobalHandleException.java để map sang HTTP status + ApiResponse. Ví dụ: AppException sẽ lấy ErrorCode và set status/code/message.

Nhóm lỗi HTTP code/message
AppException Theo exception/ErrorCode.java Map trực tiếp từ enum ErrorCode
BadCredentialsException 401 code=UNAUTHENTICATED, message “Email hoặc mật khẩu không chính xác”
AccessDenied 403 code=UNAUTHORIZED
Validation (@Valid) 400 code=INVALID_REQUEST; message tổng hợp từ field errors
Không tìm thấy endpoint 404 code=INVALID_REQUEST; message “Không tìm thấy endpoint: ...”
Lỗi hệ thống khác 500 code=UNCATEGORIZED_EXCEPTION; message “Đã xảy ra lỗi hệ thống”

Bảo mật (Security/JWT)

1) Tổng quan cơ chế

Loại auth
Bearer JWT (OAuth2 Resource Server) - stateless.
Algorithm
HMAC HS512.
Logout
Thu hồi token theo jti (denylist) đến khi token hết hạn.
Cấu hình
config/SecurityConfig.java

Chi tiết flow xác thực

Flow này giúp bạn hiểu rõ “ai verify token”, “verify ở đâu”, và vì sao logout có tác dụng ngay lập tức.

Bước Mô tả Liên quan
1 Client gọi /api/auth/login hoặc /api/auth/register để lấy token. controller/authentication/AuthController.java
2 Token được phát bằng HS512, có jti và scope. util/JWTHelper.java
3 Client gửi request kèm header Authorization: Bearer <token>. Frontend/Client
4 Spring Security Resource Server decode/verify token bằng JwtDecoder: check chữ ký, check exp (timestamp), check denylist theo jti. config/SecurityConfig.java, config/JtiDenylistValidator.java
5 Authorities được lấy từ claim scope (chuỗi cách nhau bởi dấu cách). Prefix authority được đặt rỗng, nên hệ thống dùng trực tiếp các giá trị như ROLE_ADMIN, PERMISSION_X. config/SecurityConfig.java
6 Logout: client gọi /api/auth/logout. Server lấy token từ header hoặc body, parse ra jti và lưu vào bảng denylist (InvalidatedToken). Các request sau sẽ bị chặn ngay. controller/authentication/AuthController.java

2) Public endpoints

Một số endpoint được mở không cần token: login/register/introspect/logout, eKYC, actuator health/info, swagger. Danh sách khai báo ở config/SecurityConfig.java.

3) JWT claims (token chứa gì)

Token được phát bởi util/JWTHelper.java với các claim tiêu biểu:

  • sub: email người dùng
  • iss: team2.com
  • exp: thời hạn (dựa trên jwt.valid-duration)
  • jti: mã token để revoke khi logout
  • scope: gồm ROLE_* và permission codes (cách nhau bằng dấu cách)
  • role, citizenId, collectorId, enterpriseId: thông tin hỗ trợ phân quyền/định tuyến nghiệp vụ

4) Denylist validator

Khi API nhận request, config/JtiDenylistValidator.java sẽ chặn nếu jti đã bị revoke. Trường hợp chưa đăng nhập/invalid token sẽ trả JSON theo config/JwtAuthenticationEntryPoint.java.

Gợi ý vận hành

1) Các URL hữu ích khi chạy local

URL Mục đích
http://localhost:8080/swagger-ui.html Swagger UI để test API (đăng nhập lấy token rồi Authorize).
http://localhost:8080/v3/api-docs OpenAPI JSON.
http://localhost:8080/actuator/health Kiểm tra tình trạng app.

2) Checklist cấu hình tối thiểu để chạy ổn

  • DB MySQL chạy được và set đúng SPRING_DATASOURCE_URL/USERNAME/PASSWORD
  • Set JWT_SIGNER_KEY đủ dài (HS512 yêu cầu secret đủ mạnh)
  • Nếu dùng upload ảnh: set CLOUDINARY_CLOUD_NAME/API_KEY/API_SECRET
  • Nếu dùng eKYC: set VNPT_EKYC_ACCESS_TOKEN, VNPT_EKYC_TOKEN_ID, VNPT_EKYC_TOKEN_KEY
  • Nếu muốn test nhanh: bật APP_SEED_ENABLED=true để tạo dữ liệu demo

3) Các điểm cần chú ý

  • Giữ UTC cho DB và API để tránh lệch giờ khi deploy nhiều môi trường.
  • Nếu frontend chạy domain khác, cần thêm vào app.cors.allowed-origins.
  • Profile prod đặt ddl-auto=validate, cần migration/schema đúng trước khi chạy.
Tài liệu này được tạo để đọc offline trong IDE hoặc mở trực tiếp bằng trình duyệt.
